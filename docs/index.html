Introduction

What is TaskFlow?

Think of TaskFlow as a post office for your application. Instead of processing slow tasks immediately (sending emails, crunching data, calling webhooks), you drop them into a queue and let workers handle them in the background.

📥

Submit a Task

Your app sends a job to the API. The API saves it and drops it into a queue. Your app gets an instant response — no waiting around.

⚙️

Workers Pick It Up

One of many workers grabs the job from the queue and processes it. If it fails, it automatically retries with increasing delays.

📈

Monitor Everything

The dashboard shows you exactly what’s happening: which jobs are running, which failed, how your queues are doing, and which workers are online.

States

Job state machine

Every job moves through these states. Failed jobs are automatically retried; permanently failed ones are “dead-lettered” for manual review.

Waiting ➔ Active ➔ Completed ✔
Happy path: job succeeds on first try
Active ➔ Failed 🔄 Waiting (retry)
Retry: failed job goes back to the queue with exponential backoff
Failed ➔ Dead 💀
After all retries exhausted (default 3), the job is dead-lettered
Waiting ➔ Cancelled
You can cancel any waiting job via the API or dashboard
Reliability

Exponential backoff retries

When a job fails, it doesn’t immediately try again. Instead, it waits longer and longer between attempts, giving temporary issues (like network blips) time to resolve.

➔

⏳

+2s wait

Formula: delay = 1000ms × 2attempt  —  1s → 2s → 4s → 8s …

API

REST API reference

All endpoints live under http://localhost:3001/api/v1

📥 Jobs
POST /jobs Submit a new job
GET /jobs List jobs with filters (status, queue, priority, pagination)
GET /jobs/:id Get job details
DEL /jobs/:id Cancel a waiting job
POST /jobs/:id/retry Retry a failed or dead job
📊 Queues
GET /queues List all queues with stats
GET /queues/:name/stats Detailed statistics for a specific queue
⚙️ Workers
GET /workers List all registered workers with heartbeat status
⏰ Schedules
POST /schedules Create a recurring schedule (cron)
GET /schedules List all schedules
PATCH /schedules/:id Enable or disable a schedule
DEL /schedules/:id Delete a schedule
💚 Health
GET /health Check API, Redis, and database connectivity
Getting Started

Up and running in 4 steps

Everything runs locally. You just need Node.js 20+ and Docker installed.

1

Install dependencies

Clone the project and install all packages across the monorepo.

cd taskflow
make install
2

Start infrastructure

Spin up PostgreSQL and Redis in Docker containers.

make infra
3

Set up the database

Copy the environment file and run database migrations to create tables.

cp .env.example .env
make migrate
4

Start development

This launches the API, workers, scheduler, and dashboard all at once.

make dev

# Now open:
# Dashboard → http://localhost:3000
# API → http://localhost:3001/api/v1/health

💫 Try it out!

Submit your first job with a single curl command:

curl -X POST http://localhost:3001/api/v1/jobs \
  -H "Content-Type: application/json" \
  -d '{"queue":"email","name":"send-welcome",
      "data":{"to":"user@example.com","subject":"Welcome!"},
      "priority":"high"}'

Then head to http://localhost:3000 to watch it process in real time.

Docker

Run everything in containers

For a production-like setup, use Docker Compose to run the entire stack in isolated containers.

📦 Start full stack

Builds and starts all services: Postgres, Redis, API, workers, scheduler, and dashboard.

make docker-up

🚀 Scale workers

Need more processing power? Scale workers to any count with a single command.

docker compose up --scale worker=5 -d
